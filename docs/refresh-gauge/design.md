# リフレッシュゲージ計算機 — 設計

単独の5つ目ツール（`/refresh`）。イベランの第3制約「リフレッシュゲージ」を可視化し、
**休憩を織り込んだ周回プラン全体**を出す。仕様の根拠は [measurements.md](./measurements.md)。

## ゴール（MVP＝休憩込み周回プラン全体）

入力（選曲・現在ゲージ・セッション条件）から次を出す：

1. **ゲージ推移タイムライン**：時間軸でゲージ%。100%（続行不可）到達点をマーク。
2. **休憩プラン**：100%到達後、何分休めば再開できるか。持続走行の run/rest サイクル。
3. **総獲得ポイント**：セッション全体で稼げるポイント（強制休憩の空白込み）。
4. **選曲比較**：「100%までに稼げる総ポイント」で曲を比較（短い曲がゲージ効率で有利）。

## モデル（2026-03-10 仕様・日付つき定数）

```
GAUGE_MAX          = 6_600_000        // 画面100%
LIVE_COEF          = 157              // ライブ1回の内部増加 = 定数 × 157
MYSEKAI_MATERIAL   = 700              // 通常素材 スタミナ1あたり(内部) ※157化据え置きは要再確認
MYSEKAI_FUTABA     = 250              // 双葉1本あたり(内部)  ※同上
DECAY_PER_30MIN    = 550_000          // 非活動30分ごと減少 → 8.333%/30分、100%→0%で6時間
```

- ライブ1回の表示% = `定数 × 157 / 6_600_000 × 100`（小数第2位切り捨て）
- 100%で イベントP・バッジ獲得停止。100%未満に下がれば再開（＝休憩が要る）。
- **係数は変更履歴あり（2/28→3/10）。定数表・係数はバージョン管理必須。**

## リフレッシュ定数の出し方（研究で確定）

`定数` は music_time から一意に出ない。**基礎点が唯一の厳密アンカー**：

```
定数レンジ = [ ceil(3.5×基礎点 − 267),  floor(3.5×(基礎点+1) − 267) ]   // 3〜4値
既定 = レンジ中央値（実測8曲で軌道全一致・検証済み）
上書き = 実測が取れた曲だけ measured テーブルで固定（エビ=84 等）
```

基礎点は既存の `useAnalyzerMusics`（event_rate＋verifiedBasePoints 補正）を流用。

## モジュール構成（src/pages/refresh/）

- `lib/refreshConstant.ts` … 基礎点→定数レンジ・中央値、measured 上書き、`getRefreshConstant(basePoint, id)`
- `lib/gaugeModel.ts` … 定数・純関数（1回増加・100%到達回数・減少・マイセカイ）。仕様非依存の核
- `lib/sessionPlanner.ts` … 時間モデル込みのセッション模擬（run/rest サイクル・総ポイント）
- `RefreshGaugeCalculator.tsx` … UI（SongSearchModal 流用・ニューモーフィズム）

## 時間モデル（確定・実測較正）

1周回 = 曲の長さ + オーバーヘッド（協力ライブのマッチング・結果画面・支援者交代・ロード）。
**@Nori 実測（エビ74.8s）**: 理論 32回/h（OH約38s）／実測平均 28回/h（OH約54s）。環境で大きく変わる。

- プリセット `theoretical=38 / realistic=54`、既定は realistic。
- **実測レートからOHを逆算して較正**（`overheadFromRate`）。UIで「実測レート計測」機能を持ち、
  プレイ数×経過時間から個人の回/時を出して全曲に適用する。
- ゲージ制約込みの持続ペース：エビ実測28/h → **持続約21/h（実時間の約25%が休憩）**。

## Discord bot（並行ワークストリーム）

全ツール（ゲージ／ポイント調整／実効値／ついぼ文面）を slash コマンド化。
**純ロジックは共有コアとして流用**（gaugeModel・sessionPlanner・analyzer calc・evcLogic・tweetLogic は
すべてフレームワーク非依存TS）。ホストは Cloudflare Workers（Interactions webhook・常駐不要）を第一候補、
無料枠機能次第で NAS も検討。
- 前提: Discord App＋トークン（Vaultwarden管理・値は扱わない）、テスト用guild。
- コード共有: PoCはコア流用（コピー）、定着したら shared package 化を検討。
