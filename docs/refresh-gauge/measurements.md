# リフレッシュゲージ 実測ログ（ツールのシードデータ）

プロセカ「イベント休憩システム＝リフレッシュゲージ」の**リフレッシュ定数**を実機計測で確定するためのログ。
計測は @Nori のイベラン中に実施（2026-07 時点）。

## 確定している式（3/10以降の現行仕様）

- 内部MAX = **6,600,000**（画面100%）
- ライブ1回の増加（内部値） = **リフレッシュ定数 × 157**
- 表示% = 内部値 ÷ 6,600,000 × 100（小数第2位切り捨て）
- 1回の表示% ≒ `定数 × 157 / 66000`（= 定数 × 0.0023788%）
- 逆算: `定数 = Δ表示% × 420.38 ÷ 回数`
- 減少（**公式スライド 2026-03 で確定**）:
  - 最終プレイから**5分後に計測開始**。その5分も計測に含む（実質「最終プレイから30分で初回減少」）。
  - **30分ごと −550,000（8.33%）**。100%→0% で6時間。
  - **減少途中にプレイするとタイマー停止・残り時間は繰り越し**（30分ゼロからやり直しにならない）。
    → プランでは休憩進捗をブロック跨ぎで繰り越す（20分休憩→プレイ→20分休憩＝計40分で1回減少）。
  - 未モデル化: 5分未満の休憩（プレイ間5分以内）の扱い。実運用で稀なため all-count 近似。
- マイセカイ: 通常素材 700/スタミナ1、双葉 250/本（内部値・要再検証）。採取も計測対象（素材ドロップから5分）。

## リフレッシュ定数の絞り込み方法

`定数` は music_time から一意に出ない（オフセットが曲ごとに 6〜9 でブレる）。
**厳密なアンカーは基礎点**（記事4曲で完全一致）:

```
基礎点 = floor((定数 + 267) / 3.5)
→ 逆算: 定数 ∈ [ ceil(3.5×基礎点 − 267),  floor(3.5×(基礎点+1) − 267) ]  （3〜4値のレンジ）
```

このレンジと、単発プレイの観測増分（表示%の帯から算出）の**積集合**で定数を 1〜2 値に絞る。
`長さ+9` は参考の点推定（±2〜3 ズレる）。

（計8曲、予測累積が表示%と全一致。基礎点由来モデルの妥当性は確定。）

> **結論①: `長さ+9` は不採用。** あいのうた/だんだんで過大、私は雨で過小にブレた。
> **定数の厳密アンカーは基礎点のみ**（`定数 ∈ [ceil(3.5×基礎点−267), floor(3.5×(基礎点+1)−267)]`）。
> 単発ランダムプレイは切り捨てで粗く、レンジ内1値の確定には**同一曲の連続10回計測**が必要。
>
> **結論②: モデルは軌道全体で検証済み。** 基礎点由来の定数（レンジ中央値）で累積を計算すると、
> エビ57回→あいのうた→ガーネット→だんだん早くなる→私は雨 の **5曲すべてで表示%が一致**
> （11.3→11.6→12.0→12.3→12.6）。ツールに必要な精度は基礎点だけで確保できる。
> 定数の実測上書きは精度の上積み（±1定数≒フルランで±1〜2%）で、必須ではない。
